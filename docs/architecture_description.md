# Architecture Description

## Overview

The e-commerce checkout system follows a microservices architecture with event-driven communication, built on AWS managed services. The system handles a single responsibility — processing a customer's shopping cart through to delivery — and is decomposed into eight independently deployable services: Order, Inventory, Pricing, Invoice, Payment, Shipment, Notification, and Admin. Each service owns its domain logic and data, communicating with other services exclusively through domain events rather than direct API calls. This loose coupling means any service can be updated, scaled, or replaced without cascading changes across the system.

## Event-Driven Communication

Amazon EventBridge serves as the central nervous system. When a service completes its work — for example, the Payment Service successfully charges a customer — it publishes a domain event (`payment.succeeded`) to the EventBridge bus. EventBridge routes that event to the appropriate consumers based on declarative rules. Between EventBridge and each consuming Lambda, an SQS queue provides buffering, retry semantics, and a dead-letter queue for messages that fail processing after three attempts. This architecture gives us at-least-once delivery with built-in backpressure: if the Shipment Service is overwhelmed, messages simply accumulate in its queue rather than causing upstream failures. Every consumer handles duplicate events via an idempotency check against a `processed_events` collection in MongoDB, so the at-least-once guarantee never produces incorrect side effects.

## Request Flow and Synchronous Boundary

The only synchronous interaction in the system is the initial `POST /v1/orders` call. The API Gateway validates the request, passes it to the Order Service Lambda, which performs input validation, checks for duplicate submissions via an idempotency key, persists the order to MongoDB, and returns a `201 Created` with the order ID — all within a two-second target. Everything after that point is asynchronous. The order flows through inventory verification, price calculation, invoice generation, payment processing, shipment creation, and customer notification as a chain of events, each step triggered by the completion of the previous one. This means the customer gets an immediate response while the heavy lifting — charging cards, calling shipping APIs, rendering invoices — happens in the background without blocking.

## Provider Abstractions

The system is designed to avoid vendor lock-in for third-party integrations. Payment processing, shipping, and notifications each use a strategy pattern — a common interface that concrete providers implement. The Payment Service doesn't know whether it's talking to Stripe or PayPal; it calls `charge()` on whichever provider was specified in the order. Adding a new payment provider means implementing the `PaymentProvider` interface and registering it in the provider registry — no changes to the orchestration logic, the event schema, or any other service. The same pattern applies to shipping (FedEx, DHL) and notification channels (email via SES today, SMS or push notifications tomorrow). The provider to use is determined at order creation time based on the customer's request, making it a runtime decision rather than a deployment-time configuration.

## Resilience and Error Handling

The system treats failure as a normal operating condition rather than an exception. Payment failures put the order into a `payment_failed` held state and notify the customer — there is no automatic retry, because retrying a declined card rarely succeeds and risks confusing the customer. Shipping provider outages are handled differently: the Shipment Service uses a circuit breaker with exponential backoff, retrying automatically up to a configurable threshold. If the circuit opens (the provider is consistently unavailable), the shipment is held and support staff are notified for manual intervention via the Admin API. At the infrastructure level, every SQS queue has a dead-letter queue that captures messages that couldn't be processed after three attempts, with CloudWatch alarms ensuring these don't go unnoticed. The combination of idempotent operations, circuit breakers, DLQs, and explicit held states means the system degrades gracefully — no order is silently lost, and every failure is either automatically recovered or surfaced for human action.

SQS delivers messages at least once, so every consumer must handle duplicates. The approach is: execute the idempotent operation first, then record the event ID in a `processed_events` collection. This ordering is critical — if the record were written first and the Lambda crashed before completing the work, the retry would see the event as "already handled" and skip it, silently losing the event. By processing first and marking after, the worst case on a crash is that the operation runs twice, which is safe because every operation is designed to be idempotent: MongoDB status transitions use conditional updates that only match the expected prior state, and external API calls (Stripe, FedEx) carry idempotency keys that the providers enforce. The `processed_events` lookup is purely a performance optimization — it avoids redundant calls to external services — while correctness rests entirely on operation-level idempotency.

## Data Architecture

All services share a single MongoDB Atlas cluster, but each service operates on its own collection with clear ownership boundaries. The Order Service writes to `orders` and `order_idempotency`; the Payment Service writes to `payments`; the Shipment Service writes to `shipments`, and so on. No service reads or writes another service's collection directly — cross-service data flows through events. MongoDB's document model is a natural fit here: orders embed customer details, line items, and status history in a single document, avoiding the join complexity of a relational model while keeping reads fast. TTL indexes handle automatic cleanup of ephemeral data like idempotency keys (5-minute expiry) and processed event records (24-hour expiry), keeping the database lean without manual maintenance.
